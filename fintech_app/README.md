# fintech-app

A demo personal-finance API: accounts, transactions, and a rule-based
categorization engine.

## Safety: demo data only, no real bank -- ever

**This app has never connected, and will never connect, to a real bank,
payment processor, or brokerage.** Every account and transaction in this
app is either:

- entered manually through the API (`POST /api/accounts`,
  `POST /api/accounts/<id>/transactions`), or
- generated by `mock_data.py`'s seeded synthetic data generator via
  `POST /api/demo/seed`.

There is no Plaid/bank-aggregator integration, no OAuth to a financial
institution, and no code path that could reach one. This is a hard rule
for this arc, not just a milestone-1 gap to fill in later.

## What's in milestone 1

- **Data model** (SQLite): `users`, `accounts` (checking / savings /
  credit_card), `transactions`. Account balances are never stored --
  always derived by summing the account's transactions, so they can't
  drift out of sync.
- **Categorization engine** (`categorization.py`): a pure, stateless,
  rule-based classifier. Given a merchant, description, and signed
  amount, it matches against ~90 keywords across 12 categories
  (income, transfer, housing, subscription, utilities, health,
  transport, groceries, dining, entertainment, fees, shopping) in a
  fixed priority order, falling back to `income`/`other` based on the
  amount's sign only when nothing matches. Every transaction is run
  through this engine unless the caller supplies an explicit category,
  and a manual category is sticky -- `POST /api/demo/recategorize`
  (re-running the engine, e.g. after a keyword-rule change) skips any
  transaction a human has already fixed by hand.
- **Mock data generator** (`mock_data.py`): a seeded synthetic
  accounts + several months of realistic transactions per account,
  including recurring monthly bills (rent, a subscription, a utility
  bill) planted on a fixed day-of-month -- there to give milestone 2's
  planned recurring-bill detector real signal to find. Deterministic
  *for a given seed and date*: the same `(seed, today)` pair always
  produces byte-identical output (verified by
  `test_mock_data.py::test_generate_mock_transactions_is_deterministic`),
  but since the date range is always "N months back from today," the
  actual calendar dates -- and therefore the exact transaction list --
  shift day to day. `POST /api/demo/seed` does not pin `today`, so
  reseeding on a different day produces a data set anchored to the new
  date, not a byte-identical replay of the old one.
- **Auth**: bcrypt password hashing + HS256 JWT, same shape as this
  repo's other arcs (`sub` claim always encoded as a string per
  `dev_collab_platform`'s PR #11 fix).

## API

All routes except `/api/signup`, `/api/login`, and `/api/health`
require `Authorization: Bearer <token>`.

| Method | Path | Description |
| --- | --- | --- |
| POST | `/api/signup` | Create a user, returns a token |
| POST | `/api/login` | Returns a token |
| GET | `/api/accounts` | List the caller's accounts (with derived balance) |
| POST | `/api/accounts` | Create an account manually |
| GET | `/api/accounts/<id>` | Get one account |
| DELETE | `/api/accounts/<id>` | Delete an account (cascades transactions) |
| GET | `/api/accounts/<id>/transactions` | List an account's transactions (`?category=`, `?start_date=`, `?end_date=`) |
| POST | `/api/accounts/<id>/transactions` | Add a transaction (auto-categorized unless `category` is supplied) |
| GET | `/api/transactions` | List all of the caller's transactions across accounts |
| GET | `/api/transactions/categories` | List every known category |
| GET | `/api/transactions/<id>` | Get one transaction |
| PATCH | `/api/transactions/<id>` | Recategorize a transaction (sticky) |
| DELETE | `/api/transactions/<id>` | Delete a transaction |
| POST | `/api/demo/seed` | Generate 3 mock accounts + synthetic transactions (`{"months": 1-12, "reset": bool}`) |
| POST | `/api/demo/recategorize` | Re-run the categorization engine over non-manually-set transactions |

## Running it

```bash
pip install -r ../requirements.txt
python run.py   # serves on http://127.0.0.1:5060
```

```bash
curl -X POST localhost:5060/api/signup \
  -H 'Content-Type: application/json' \
  -d '{"email":"me@example.com","password":"password123"}'
# grab the token from the response, then:
curl -X POST localhost:5060/api/demo/seed \
  -H "Authorization: Bearer <token>" -H 'Content-Type: application/json' -d '{"months":3}'
curl localhost:5060/api/transactions -H "Authorization: Bearer <token>"
```

## Tests

```bash
pytest fintech_app/tests -q
```

55 tests: categorization rules (priority order, fallback behavior,
case-insensitivity), mock-data determinism, auth, account/transaction
CRUD and ownership isolation (404-not-403 on another user's data), and
the demo-seed/recategorize endpoints.
